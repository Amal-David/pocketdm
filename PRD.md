## Amendments (v1.1, June 10, 2026)

Decisions made after research on Day 1; these override the body text below wherever they conflict.

1. **Student model = Qwen3.5-2B** (Apache 2.0), replacing Gemma-3-270M, which was judged unworkable. "Qwen 4" does not exist; Qwen3.5 (Feb 2026) is the newest small generation. New ladder in §5.1. Marketing line becomes *"a Dungeon Master in 1.2GB, with the Wi-Fi off."*
2. **Ship Q4_K_M GGUF, not Q8_0** — Q8 roughly halves CPU tok/s for marginal quality on a narrow fine-tuned task.
3. **Latency contract is streaming-based** (first token ≤2s, ≥10 tok/s sustained), not "≤4s full turn" — that figure assumed a 270M model. Hard gate: <10 tok/s measured on a real free-tier Space → retrain on Qwen3.5-0.8B.
4. **Teacher = Qwen3-32B** (Apache 2.0, single-H100 BF16, vLLM xgrammar structured output). Llama-3.3-70B is explicitly avoided: its license forces models trained on its outputs to carry "Llama" in the name. → Open Question 2 resolved.
5. **4th voice added: "Lore Narrator"** — a custom-blended Kokoro woman voice in dark-fantasy game-lore narration style (blend of bf_emma / af_heart / af_nicole + slow pacing + reverb DSP), selectable in any genre. Kokoro has no public training code, so this is a voice-tensor blend, described honestly as such; the Well-Tuned claim rests solely on the DM model. Never built from game audio (copyright / right-of-publicity).
6. **UI mechanism confirmed**: `gr.Server` (Gradio ≥ 6.10) is real and explicitly named by the Off-Brand badge. Pin `sdk_version` ≥ 6.10 in the Space README. → Open Question on frontend resolved.
7. **Licensing is Apache 2.0 end-to-end** (student, teacher, Kokoro) → Open Question 1 resolved/moot.
8. Total runtime parameter budget: 2B (DM) + 82M (Kokoro) ≪ the 32B cap, inside Tiny Titan's ≤4B.

## 1. Problem & Opportunity

Tabletop-style interactive fiction is the perfect showcase for a tiny model — short creative bursts, a constrained format, and a deterministic game engine that carries the logic. Yet the hackathon field (244+ Spaces) is crowded with multi-agent economies and document-triage assistants, while almost nobody has shipped (a) a genuinely tiny fine-tuned model as the star of the show, (b) a voice-native experience, or (c) a custom `gr.Server` frontend with real production polish.

PocketDM occupies all three open lanes at once. The thesis — and the demo's closing line — is: **"Everything you just played was generated by a model that fits in 1.2GB, with the Wi-Fi off."**

### Why this wins
The hackathon explicitly stacks awards. One well-executed Space can simultaneously earn the Tiny Titan award (model ≤4B), the Well-Tuned badge (your own published fine-tune), Off the Grid (no cloud APIs at inference), Llama Champion (llama.cpp runtime), Off-Brand (custom UI, $1,500 award), Best Demo, a Thousand Token Wood podium, plus the Codex and Modal sponsor prizes. No other idea in our research stacks this many with this little compute risk.

## 2. Goals & Non-Goals

### Goals
1. **G1 — Win.** Maximize stacked prize eligibility; everything below is in service of this.
2. **G2 — Tiny model, load-bearing AI.** The fine-tuned ≤4B model generates every word of narration and every choice. No hidden large model at inference time.
3. **G3 — Fully offline inference.** llama.cpp GGUF + CPU TTS; zero network calls during play. Provable on camera.
4. **G4 — Voice-native.** The DM *speaks* every scene (Kokoro-82M TTS).
5. **G5 — Demo-first polish.** A 10-turn session that never breaks, inside a custom parchment UI that looks nothing like stock Gradio.
6. **G6 — Codex-authored.** All production commits attributed to Codex in a public GitHub repo, with published agent traces.

### Non-Goals (cut ruthlessly)
- ❌ Multiplayer, accounts, or cross-session persistence (one browser session = one adventure).
- ❌ Generated images (adds a model, latency, and risk; use hand-picked static genre art instead).
- ❌ Long campaigns (>20 turns), combat math systems, or character sheets beyond HP + inventory.
- ❌ Speech-to-text input (typing/clicking is fine; STT adds latency and failure modes).
- ❌ Beating GPT-class quality. The pitch is *capability-per-parameter*, not absolute quality.

## 4. Product Requirements

### 4.1 Core gameplay loop (P0)

- **FR-1:** Player picks a **genre** (3 launch genres: 🏰 Cursed Dungeon, 🍄 Whispering Wood, 🚀 Derelict Starship) and optionally types a one-line premise ("I'm a goblin chef who stole the king's spoon").
- **FR-2:** Each turn, the model emits **one structured turn object** (schema below). The engine renders: 2–4 sentences of narration, exactly **3 choices**, and any state changes.
- **FR-3:** Player advances by clicking a choice or typing a freeform action (freeform is injected as the chosen action).
- **FR-4:** Game state = `{hp: 0–10, inventory: string[≤6], location: string, flags: string[], turn_count}`. State is owned by a **deterministic Python engine**, never by the model. The model proposes deltas; the engine validates, clamps, and applies them.
- **FR-5:** Adventures resolve in **8–15 turns**. The engine injects escalating "story pressure" hints into the prompt (act 1 → act 2 → finale) and forces `is_ending: true` by turn 15. HP 0 triggers a death ending (which should be funny, not bleak).
- **FR-6:** A turn must **never** render broken. JSON validity is guaranteed by grammar-constrained decoding (see 5.4); if generation stalls or content fails engine validation, retry once with higher temperature, then fall back to a genre-appropriate canned "the fog thickens…" bridge turn. Target: **0 visible failures in any 15-turn session.**

**Turn schema (the contract everything is built around):**

```json
{
  "narration": "2-4 sentences, second person, present tense",
  "choices": ["short imperative", "short imperative", "short imperative"],
  "state_delta": {
    "hp": 0,
    "add_items": [],
    "remove_items": [],
    "location": "current or new location name",
    "add_flags": []
  },
  "is_ending": false,
  "ending_type": null
}
```

### 4.2 Voice (P0)

- **FR-7:** Every narration is spoken aloud via **Kokoro-82M** (Apache 2.0, CPU-capable, ~300MB). Audio autoplays as the typewriter text reveals.
- **FR-8:** Each genre maps to a distinct Kokoro voice preset (gravelly dungeon narrator, warm woodland storyteller, clipped ship-AI), **plus a 4th selectable "Lore Narrator"** — a custom-blended Kokoro woman voice in dark-fantasy narration style (Amendment 5), usable in any genre.
- **FR-9:** Mute toggle; TTS generation must not block text rendering (generate audio async, attach when ready). TTS runs strictly sequentially after decode on the 2-vCPU Space (never concurrent with token generation).

### 4.3 Custom UI — the Off-Brand play (P0)

- **FR-10:** Built as a **custom `gr.Server` frontend** (HTML/CSS/JS served by Gradio ≥ 6.10) — *not* themed default components. Visual identity: aged parchment scroll on a dark wooden desk, ink-style typewriter text, wax-seal choice buttons, hand-drawn HP hearts and inventory icons in a margin sidebar.
- **FR-11:** Persistent **proof-of-tininess badge** in the corner: `2B params · 100% offline · llama.cpp` with a live tokens/sec readout. This badge is the brand.
- **FR-12:** Micro-delights: d20 dice-roll animation between turns, page-turn transition on location change, subtle ambient loop per genre (toggleable), confetti of tiny dice on a good ending.
- **FR-13:** Mobile-responsive (judges and voters will open it on phones).
- **FR-14:** "📜 New Adventure" always one click away; a session must be startable in <5 seconds from page load (after Space warm-up).

### 4.4 Offline proof & deployment (P0)

- **FR-15:** Inference stack inside the HF Space: **llama.cpp** running the fine-tuned model as **Q4_K_M GGUF on CPU** + Kokoro on CPU. **Zero external API calls at inference** (audit with a network-call assertion in CI). This earns Off the Grid + Llama Champion.
- **FR-16:** Ship a `run_local.sh` / `uv run pocketdm` one-liner so anyone can download the GGUF and play on a laptop with no internet — and so the demo video can show airplane-mode gameplay on your own machine.
- **FR-17:** Space lives in the hackathon org with README frontmatter declaring track + all badge tags, demo video link, and social post link (hard submission requirements).

### 4.5 Submission artifacts (P0 — these are scored deliverables, not chores)

- **FR-18:** Published **model repo** on HF (safetensors + GGUF, full model card with eval table) → Well-Tuned.
- **FR-19:** Published **dataset repo** (the synthetic DM turns) + **agent/training traces** → Sharing is Caring.

## 5. ML Requirements

### 5.1 Model ladder (amended v1.1 — decided June 10)

| Rung | Model | Why / when |
|---|---|---|
| **Primary** | **Qwen3.5-2B, fine-tune (full FT preferred, LoRA fallback)** | Apache 2.0, newest small generation, IFEval 61.2, non-thinking by default (clean grammar-constrained JSON), unambiguously ≤4B. Q4_K_M ≈ 1.2GB. |
| Fallback 1 | Qwen3.5-0.8B, full FT | Latency insurance: triggered if the 2B measures <10 tok/s on a real free-tier Space. Same license/pipeline/recipe. |
| Fallback 2 | Gemma 4 E2B (BF16 FT → Q4 GGUF) | Quality rung. Blocked on organizers confirming its 5.1B total / 2.3B effective params pass the ≤4B Tiny Titan cap, and on acceptable CPU latency (~7–12 tok/s measured on Pi-5-class hardware). |

**Gate rule:** demotion to 0.8B is triggered by *measured latency*; promotion to E2B by *measured quality* (eval harness coherence <3.5/5 after one data-quality iteration) — both measured, never vibed. Max half a day fighting any rung.

Total runtime parameter budget: 2B (DM) + 82M (Kokoro) ≈ **2.1B** — comfortably under the 32B cap and the Tiny Titan ≤4B line. Put this math in the README.

### 5.2 Training data — synthetic distillation (Day 1)

- **DR-1:** Teacher = **Qwen3-32B served via vLLM batch on Modal** (Apache 2.0; uses Modal credits → Best Use of Modal evidence; avoids API-ToS gray zones and the Llama output-naming clause; fully reproducible/publishable).
- **DR-2:** Generate **~1,500–2,500 complete adventures ≈ 15–25k turns** across the 3 genres + diverse premises, *in the exact turn schema*, by having the teacher simulate both DM and a player policy (mix of greedy/curious/chaotic player personas so choice-taking is diverse).
- **DR-3:** Every training example = `(system prompt + compact state summary + last 2 turns) → (turn JSON)`. The model is trained to see **engine-rendered state, not raw history** — this is what keeps a small model viable.
- **DR-4:** Quality gates before training: schema-valid (drop invalid), narration 2–4 sentences, choices mutually distinct (embedding cosine <0.85 pairwise), no choice repeats from prior turn, deltas legal for current state, profanity/content filter. Target **≥95% pass rate post-filter; quality > quantity** (a clean 10k beats a noisy 25k).
- **DR-5:** Hold out 100 full adventures as the eval seed set.
- **DR-6:** Estimated cost: ~10–15M output tokens of batch generation ≈ **$5–10 on an H100 with vLLM** (xgrammar structured output; watch guided-decoding throughput at high batch sizes).

### 5.3 Training (Day 2)

- **TR-1:** Fine-tune Qwen3.5-2B with Unsloth on Modal (full FT on A100 preferred — single-digit dollars; LoRA per Unsloth's Qwen3.5 recipe as fallback). 2–3 epochs, cosine schedule, packed sequences, completion-only loss on the JSON target, thinking mode off.
- **TR-2 (stretch, only if Days 1–2 go clean):** a short **GRPO pass (~300 steps, ~$1–2)** with verifiable rewards: schema validity *without* grammar assistance, choice-distinctness, delta legality, narration length window.
- **TR-3:** Export **Q4_K_M (ship) and Q8_0 (local/comparison)** GGUF — Q8 halves CPU tok/s for marginal quality on this narrow task.
- **TR-4:** Track every run cost; publish the total in the blog.

### 5.4 Inference reliability (P0 — the make-or-break requirement)

- **IR-1:** **GBNF grammar** in llama.cpp pinned to the turn schema → structurally invalid JSON becomes impossible. The fine-tune's job is making the *content* good; the grammar's job is making the *format* certain. Hand-write the GBNF (no `\d`-style regex shorthands); pin a **fresh llama.cpp build** (Qwen3.5's hybrid Gated-DeltaNet arch support is recent); grammar does not constrain EOS — pair with max-tokens + retry.
- **IR-2:** Sampling: temp ~0.8, repetition penalty ~1.15, top-p 0.95; seed per session for reproducible bug reports. Thinking mode stays off (Qwen3.5 small default).
- **IR-3:** Engine-side semantic validation (choices distinct, delta legal, no verbatim repetition of last narration) → one retry → canned bridge turn (FR-6).
- **IR-4 (amended):** Latency contract on Space CPU is **streaming-based**: first token ≤2s, ≥10 tok/s sustained so the typewriter never stalls; TTS ready ≤3s after text completes. Hard gate: <10 tok/s measured → drop to Qwen3.5-0.8B.

### 5.5 Eval harness (Day 2, reused in blog + model card)

- **ER-1:** Automated: % schema-valid without grammar, choice-distinctness rate, delta-legality rate, 15-turn completion rate without fallback, mean tokens/turn.
- **ER-2:** LLM-judge rubric (run as Modal batch) on 50 held-out seeds: coherence, choice meaningfulness, ending satisfaction — 1–5 each. **Ship gate: coherence ≥3.5, zero-fallback sessions ≥90%.**
- **ER-3:** Publish a **base vs. fine-tuned** table in the model card (the before/after is the Well-Tuned story).

## 6. Architecture

```
┌─ Build time (Modal, ~$10–25 total) ─────────────────────────────┐
│ vLLM batch: Qwen3-32B teacher → synthetic adventures (~$5–10)   │
│ Quality filters + dedupe (CPU, ~$0)                             │
│ Unsloth FT Qwen3.5-2B (A100, ~$1–3) [+ GRPO ~$2 stretch]        │
│ Eval harness batch (LLM judge, ~$2)                             │
│ → GGUF export → push model + dataset + traces to HF             │
└─────────────────────────────────────────────────────────────────┘
┌─ Run time (HF Space, $0, fully offline) ────────────────────────┐
│ gr.Server custom frontend (parchment UI, JS, @gradio/client SSE)│
│   ↕                                                             │
│ Python game engine (state, validation, story pressure, retries) │
│   ↕                                                             │
│ llama.cpp · Qwen3.5-2B-PocketDM Q4_K_M GGUF · GBNF (CPU)        │
│ Kokoro-82M TTS via kokoro-onnx (CPU, sequential after decode)   │
│   incl. custom-blended "Lore Narrator" voice artifact           │
└─────────────────────────────────────────────────────────────────┘
```

Key principle (lesson from failed submissions in this hackathon): **the model never does logic; the engine never does prose.**
